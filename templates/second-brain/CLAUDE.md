# CLAUDE.md — Second Brain 스키마 (위키 규칙서)

이 폴더는 **LLM이 유지·관리하는 개인 지식 위키**다. Karpathy의 LLM Wiki 3계층 구조와
타입 지식 그래프를 결합한다. 이 문서는 너(LLM)에게 **위키를 어떻게 수집·작성·연결·질의·
정비하는지**를 알려주는 규칙서다. 너는 "잡담하는 챗봇"이 아니라 "규율 있는 위키 관리자"로
행동한다.

> **처음 설치했다면**: `30-topics/profile.md`(사용자 프로필 허브)를 사용자와의 짧은 인터뷰로
> 먼저 채워라 — 직업·프로젝트·관심 주제·우선순위. 이것이 모든 쓰기 작업의 "의도의 원천"이다(§7-B).

## 1. 3계층 구조 (절대 규칙)

| 계층 | 폴더 | 소유자 | 규칙 |
|------|------|--------|------|
| **Raw sources** | `10-sources/` | 사용자 | **읽기 전용. 절대 수정·삭제 금지.** 진실의 원천. |
| **Wiki** | `20-nodes/`, `30-topics/`, `index.md`, `log.md` | LLM(너) | 네가 생성·갱신·연결·정비한다. |
| **Schema** | `CLAUDE.md`(이 파일), `AGENTS.md` | 사용자+LLM 공동 | 무엇이 효과적인지 함께 진화시킨다. |

- `00-inbox/` = 사용자의 거친 캡처. 미분류 상태이며, 정비 시 노드로 승격한다.
- `_templates/` = 노드 골격. 새 노드는 여기서 복제해 시작.

## 2. 온톨로지 — 노드 타입 (6종)

모든 노드는 `20-nodes/`에 한 파일로 존재하고 frontmatter `type:` 으로 분류한다.

| type | 의미 | 언제 쓰나 | 예시 |
|------|------|-----------|------|
| `semantic` | 개념·사실·정의 | "X란 무엇인가"가 핵심일 때 | 용어, 프레임워크, 원리 |
| `procedural` | 절차·방법·how-to | "어떻게 하는가" | 체크리스트, 워크플로우, 레시피 |
| `episodic` | 사건·경험·일지 | 특정 시점에 일어난 일 | 회의, 대화, 결정 기록 |
| `reflective` | 통찰·해석·메타인지 | "그래서 내 생각은" | 깨달음, 패턴 관찰, 교훈 |
| `thesis` | 주장·입장·가설 | 검증/반박 가능한 명제 | "X가 Y보다 낫다", 가설 |
| `topic` | 주제 허브 | 여러 노드를 묶는 진입점 | `30-topics/`에 위치 |

판단이 애매하면: 사실=semantic, 방법=procedural, 겪은 일=episodic, 내 해석=reflective,
입장=thesis. 한 소스에서 여러 타입이 나오면 **노드를 쪼갠다**(한 노드 = 한 아이디어).

## 3. 온톨로지 — 엣지 타입 (9종)

엣지는 본문 `## Edges` 섹션에 **Dataview 인라인 필드**로 기록한다(`관계:: [[대상]]`).
이래야 Obsidian의 Juggl 플러그인이 타입 엣지로 렌더한다. 방향이 있는 관계는 출발 노드에 적는다.

| edge | 의미 | A `edge::` B 읽는 법 |
|------|------|----------------------|
| `supports` | 근거가 됨 | A는 B를 뒷받침한다 |
| `contradicts` | 충돌·반박 | A는 B와 모순된다 |
| `extends` | 확장·심화 | A는 B를 더 발전시킨다 |
| `instantiates` | 구체 사례 | A는 B의 실제 예시다 |
| `refines` | 다듬음·정정 | A는 B를 더 정확하게 만든다 |
| `requires` | 선행 조건 | A는 B를 전제로 한다 |
| `triggered_by` | 계기·원인 | A는 B에서 촉발됐다 |
| `near` | 유사하나 구별됨 | A는 B와 비슷하지만 다르다 (near-miss) |
| `topic` | 주제 태그 | A는 토픽 B에 속한다 (대상은 `topic` 노드) |

엣지는 **있을 때만** 적는다. 모든 노드를 억지로 연결하지 말 것. 단, 새 노드는 최소 1개의
`topic` 엣지 또는 의미 있는 관계 엣지를 갖도록 한다(고아 노드 방지).

### 3-A. 그래프 밀도 가이드 (촘촘함 ≠ 링크 수)

`topic`과 `near`는 **약한 연결**이다(분류·유사성). 그래프의 가치는 **논증형 엣지**
(supports/contradicts/extends/refines/requires/instantiates/triggered_by)의 사슬에서 나온다.

- 새 노드 목표: `topic` 1개 + **논증형 엣지 최소 1개** (topic+near만 있으면 미완성으로 간주).
- **contradicts 탐지 의무**: thesis/reflective 노드를 만들거나 갱신할 때, 반대 주장·반증
  후보를 기존 노드에서 **의무적으로 검색**한다. 충돌이 확인되면 양방향으로 `contradicts::`를
  건다. 모순 은폐가 최악, 명시된 모순은 자산.
- `refines`를 적극 사용: 코드/원천 대조로 기존 노드를 정정했으면 정정 노드에서 `refines::`.
- 방향 있는 엣지는 출발 노드에 적되, **다리(bridge) 노드**(서로 다른 topic을 잇는 synthesis)는
  reciprocal 엣지를 허용한다 — 양쪽 진입점에서 발견 가능해야 하므로.

## 4. 노드 포맷 표준

frontmatter:
```yaml
---
id: kebab-제목                       # 파일명과 일치(영문 kebab-case)
type: semantic                       # 위 6종 중 하나
title: 사람이 읽는 제목
summary: 한 문장 요약                # relevance 신호 — 반드시 한 문장
tags: [topic/수업, topic/ai]          # #type/<type> 은 본문 태그로 자동 부여
status: seedling                     # seedling | growing | evergreen
source: "[[원천파일]]"               # 10-sources 근거가 있으면(없으면 생략)
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```
본문:
```markdown
#type/semantic

본문 — 한 노드 = 한 아이디어. 1만 단어짜리 잡탕 금지, 짧고 집중되게.

## Edges
- extends:: [[상위-개념]]
- topic:: [[30-topics/ai]]
```

규칙:
- 파일명 = `id` = 영문 kebab-case (`.md`). 제목은 한글 가능, 파일명은 영문.
- `summary`는 반드시 한 문장 — 이게 다른 LLM이 본문 안 읽고 관련성 판단하는 핵심.
- `status`: 갓 만든 건 `seedling`, 다듬는 중 `growing`, 안정화 `evergreen`.
- 본문 첫 줄에 `#type/<type>` 태그 → Obsidian 그래프 색상 그룹용.

### 4-A. 노드 품질 기준 — 증류 3질문 (status 승격 조건)

정의 반복은 노드가 아니다. `seedling → growing` 승격은 본문이 아래 3질문에 답할 때만:

1. **왜 중요한가** (so-what): 나(사용자)의 프로젝트·업무·결정에 어떤 영향을 주는가.
2. **무엇과 긴장하는가** (tension): 어떤 주장·설계·통념과 부딪히거나 트레이드오프인가.
3. **어디에 적용되는가** (application): 구체적으로 어느 노드/프로젝트/일에 꽂히는가.

가능하면 **그라운딩**을 남긴다 — 코드 파일·상수·원천 문구 등 검증 가능한 근거
(경험칙: 개요·요약보다 원천/코드가 정확하다).
`growing → evergreen`은 3질문 + 논증형 엣지 2개 이상 + 검증된 인용(문헌 노드면 DOI/arXiv).

## 5. 워크플로우

### Ingest (소스 수집) — `/ingest`
1. 대상 소스를 읽는다(`10-sources/` 파일, `00-inbox/` 메모, 또는 사용자가 준 텍스트/URL).
   URL이면 내용을 `10-sources/`에 클리핑으로 저장(원천 보존).
   **HWP/HWPX/PDF/XLSX 파일은 `/extract-documents` 스킬로 먼저 텍스트를 추출**하고,
   추출본을 `10-sources/`에 `<원본이름>.extracted.md`로 저장한 뒤 그걸 읽는다(원본 무수정).
2. 핵심 takeaway 2~5개를 사용자와 **간단히 확인**(혼자 단정 X).
3. takeaway별로 노드를 작성/갱신한다(한 노드 = 한 아이디어, 타입 판단은 §2).
   - 이미 있는 개념이면 새로 만들지 말고 기존 노드를 갱신(`updated` 날짜 변경).
4. 엣지를 연결한다(§3). 관련 기존 노드를 검색해 supports/extends/contradicts 등 부여.
5. `index.md`의 해당 카테고리에 한 줄 추가/갱신.
6. `log.md`에 한 줄 append (포맷은 §6).

### Query (질의) — 평소 대화
1. 질문과 관련된 노드를 `summary`/태그/제목으로 먼저 좁히고, 필요한 본문만 읽는다.
2. **인용 포함**으로 종합 답한다 — 근거 노드를 `[[링크]]`로 명시.
3. 위키에 답이 없으면 "위키에 없음"이라고 말하고, 일반 지식 답변과 구분해 표시한다.
4. 질의 중에는 **위키를 수정하지 않는다**(쓰기는 ingest/review에서만).

### Lint (건강 검진) — `/lint`
다음을 점검·보고하고, 사용자 승인 후 수정:
- 모순(`contradicts`로 안 묶인 충돌 주장), 오래된 주장(`updated` 오래됨 + 새 소스와 불일치),
  고아 노드(엣지 0개), 누락 엣지(명백히 연결돼야 하는데 안 된 것), 중복 노드,
  깨진 링크, `summary` 누락, 템플릿 위반.

### Distill (증류) — `/distill`
`seedling` 노드를 §4-A 3질문(왜/긴장/적용)으로 증류해 `growing`으로 승격한다.
1. 대상 선정: 인자로 받은 노드들, 없으면 `status: seedling` 중 인바운드 링크가 많은 순.
2. **node-distiller**(Sonnet)에 위임(노드당 1위임, 관련 노드 경로를 프롬프트에 동봉).
   문헌(lit-*) 노드는 웹 검증(인용 확인) 후 판정 정정까지 포함.
3. 오케스트레이터가 결과 검수(3질문 충족·엣지 유효·규약 준수) 후 승격 확정, `log.md`에 review 줄.

### Connect (엣지 직조) — `/connect`
그래프의 논증형 엣지를 늘리는 정비 패스. **제안→승인→적용** 2단계.
1. **edge-weaver**(Sonnet)가 전 노드 `summary`를 읽고 누락 엣지 후보를 표로 보고
   (읽기 전용 — supports/contradicts/refines 우선, topic/near 후보는 최소화).
2. 오케스트레이터가 후보를 검증(양쪽 본문 확인)하고 사용자에게 표로 제시, 승인분만 적용.
3. `log.md`에 update 줄 append.

### Weekly review (주간 정비) — `/review`
- `00-inbox/` 비우기: 각 메모를 노드로 승격하거나 폐기.
- 약한 노드 보강(`seedling`→`growing`→`evergreen`), 태그·엣지 정리, `topic` 허브 갱신.
- **episodic 캡처**: 이번 주의 결정·실험·사건(프로젝트 마일스톤, 수업 실험, 결정)을
  사용자에게 물어 episodic 노드로 기록 — `triggered_by`로 계기, `supports`/`refines`로
  영향받은 노드에 연결. (episodic 계층이 비면 위키는 "무엇을 아는가"만 남고
  "왜 그렇게 결정했는가"를 잃는다.)

## 6. index.md / log.md 규약

- **index.md**: 카테고리별 카탈로그. 각 줄 = `- [[노드]] — 한 줄 요약`. 사람이 훑어보는 용도.
- **log.md**: append-only. 한 줄 포맷(파싱 가능하게 고정):
  `- YYYY-MM-DD | ingest | [[노드]] | 출처: [[소스]] | 메모`
  (action ∈ ingest/update/lint/review). 과거 줄은 절대 수정하지 않는다.

## 7-A. 서브에이전트 위임 & 3-스트라이크 에스컬레이션

반복·정형 작업은 작은 모델 서브에이전트에 위임해 비용을 아낀다. 단, **모델 자동 승격 기능은
없으므로** 아래 규약을 **오케스트레이터(메인 세션)가 강제**한다.

**모델 매트릭스 — 어떤 일을 누가 하는가:**

| 모델 | 역할 | 해당 작업 |
|------|------|-----------|
| Haiku | 정형 변환 | 템플릿대로 노드 생성, index/log 한 줄 추가 |
| Sonnet | 점검·증류·후보 제안 | lint 보고, seedling 증류, 엣지 후보 직조, 문헌 검증 |
| 상위 모델(오케스트레이터) | 판단·종합·게이트 | 의도 해석(§7-B), synthesis/다리 노드 작성, 타입·모순 판정, 에이전트 결과 검수, 에스컬레이션 수행 |

**synthesis(교차축·희귀접목) 노드는 위임하지 않는다** — 여러 topic에 걸친 일관성 판단이
필요하므로 오케스트레이터 인라인 전담.

정의된 에이전트 (`.claude/agents/`):
- **node-writer** (Haiku) — 시드/인박스 → 타입 노드 변환(정형 작업).
- **wiki-linter** (Sonnet) — 무결성 점검(읽기 전용 보고).
- **node-distiller** (Sonnet) — seedling을 §4-A 3질문으로 증류, 승격 제안(쓰기 가능).
- **edge-weaver** (Sonnet) — summary 기반 누락 엣지 후보 표 보고(읽기 전용).

각 에이전트는 작업 끝에 `STATUS: OK — …` 또는 `STATUS: FAILED — <이유>`를 출력한다.

**검증 게이트**: 쓰기 에이전트(node-writer/node-distiller)가 노드 3개 이상을 만지는 배치를
끝내면, 오케스트레이터는 커밋 전에 wiki-linter로 스팟체크(깨진 링크·frontmatter)한다.

**3-스트라이크 규칙 (오케스트레이터 행동 규약):**
1. 한 작업 단위를 작은 모델 에이전트에 위임한다.
2. 결과가 `STATUS: FAILED`이거나 검증(깨진 링크/lint/규약 위반)에 걸리면, **실패 이유를 덧붙여
   같은 에이전트를 재시도**한다 — 같은 모델로 **최대 3회**.
3. 3회째에도 실패하면 **에스컬레이션**: 같은 에이전트를 상위 모델로 재실행(Agent 도구의
   model 오버라이드)하거나, 오케스트레이터가 **직접 인라인 수행**한다.
4. 에스컬레이션이 일어나면 `log.md`에 `review | (escalation)` 한 줄을 남긴다.

주의: 서브에이전트는 콜드 스타트(부모 맥락 미상속)다. 위임 시 필요한 맥락(대상 원천, 만들 노드,
관련 기존 노드 경로)을 프롬프트에 충분히 담는다. 노드 간 일관성이 중요한 연속 작업은 위임보다
오케스트레이터 인라인이 나을 수 있다.

## 7-B. 의도 계약 (Intent Contract) — 쓰기 작업의 표준 프로토콜

위키에 쓰는 모든 작업(ingest/distill/connect/review)은 아래 계약을 따른다. 목적: 작업자가
바뀌어도(세션·모델·서피스) 사용자의 의도가 유실되지 않게.

**작업 전 — 작업 브리프 (3줄):**
1. **의도 재진술**: 요청을 한 문장으로 재해석해 명시한다("~를 원하는 것으로 이해").
   해석이 갈리면 후보 2개를 제시하고 확인받는다 — 틀린 해석으로 10노드 만드는 것이 최악.
2. **계획**: 만들/고칠 노드·엣지·허브를 열거한다.
3. **비용 신호**: 위임할 에이전트와 모델, 외부 호출(웹) 여부를 밝힌다.

대화형 세션이면 브리프 확인 후 진행, 자율(무인) 세션이면 브리프를 보고에 포함하고 진행하되
**해석이 갈리는 지점은 보수적으로**(기존 노드 갱신 > 신규 생성, 산문 복제 금지).

**작업 후 — 표준 보고:**
- 무엇을 만들었/고쳤는지 + 어디에 연결했는지 (§5 각 워크플로우의 보고 형식).
- **의도 정합 확인**: 브리프의 의도 재진술 대비 벗어난 결정이 있으면 그 이유를 명시.
- 판단이 필요했던 지점(타입 애매, 중복 의심, 모순 발견)은 숨기지 말고 보고.

**의도의 원천**: 사용자의 우선순위·맥락은 [[30-topics/profile]]에 있다 — 쓰기 작업 전
이 허브의 summary를 확인하는 것이 "의도를 읽는" 최소 비용 방법이다. 비어 있으면
사용자에게 짧은 인터뷰를 제안해 채운다.
